# 数字员工功能说明

## 一、为什么需要数字员工

在通用 AI 助手的场景中，模型往往以「全能型」角色出现：既能写代码，也能查监控，还能做运维。这种设计在个人使用或简单任务时足够，但在**企业生产环境**中会遇到明显问题：

1. **上下文混杂**：同一会话中，通用助手需要同时理解运维、开发、监控等多种领域知识，容易导致回答泛化、不够精准。
2. **工具过载**：把所有 Skills 和 MCP 工具都暴露给一个会话，既增加 token 消耗，也容易让模型在无关工具间犹豫。
3. **角色模糊**：用户难以获得「专业角色」的对话体验，例如希望和「SRE 运维专家」对话时，期望的是专业、克制的运维风格，而非通用闲聊。
4. **Token 浪费与技能干扰**：大量 Skills 和 MCP 同时注入会显著推高 token 消耗，且模型在众多能力中易产生误匹配和决策干扰，影响响应质量。

**数字员工**通过为不同垂直场景创建**专属对话模版**，将「一个全能助手」拆分为「多个专业角色」，每个角色拥有独立的人设、**精简的垂直技能**和**专属 MCP 配置**，从而在降低 token 消耗、减少技能干扰的同时，提升专业场景下的对话质量和效率。

## 二、为什么需要更加垂直专业的角色

### 2.1 垂直专业化的价值

| 维度 | 通用助手 | 数字员工（垂直角色） |
|------|----------|----------------------|
| 人设与语气 | 通用、中性 | 可定制为 SRE、运维、开发等专业风格 |
| 技能范围 | 全局 Skills 全部可见 | 仅加载该角色相关的 Skills |
| MCP 工具 | 全局 MCP 全部可用 | 可为角色配置专属 MCP（如 Prometheus、K8s） |
| 会话隔离 | 同一会话易混杂多领域 | 每个角色独立会话，上下文更聚焦 |
| 使用心智 | 「问 AI」 | 「找 SRE 专家」「找运维助手」 |

### 2.2 垂直专业技能与 MCP 的价值

数字员工的核心优势在于**按需加载**：仅为当前角色加载与其职责相关的 Skills 和 MCP，而非「大而全」地暴露全部能力。

| 能力来源 | 通用助手 | 数字员工 |
|----------|----------|----------|
| **Skills** | 全局 20+ 技能全部注入上下文 | 仅 3～5 个垂直相关技能（如 SRE 仅加载 prometheus-analysis、kube-medic、monitoring） |
| **MCP 工具** | 所有 MCP 服务器全部连接 | 仅连接该角色所需 MCP（如 Prometheus、K8s API） |
| **领域知识** | 多领域混杂，易产生幻觉 | 单一垂直领域，知识边界清晰 |

**建议配置原则**：

- **技能精简**：每个数字员工仅配置 2～6 个与该角色强相关的 Skills，避免「堆砌」。
- **MCP 最小化**：只为该角色实际会用到的数据源/工具配置 MCP，减少无关工具对决策的干扰。
- **人设聚焦**：Prompt 中明确该角色的职责边界，引导模型「只做分内之事」，不越界调用无关能力。

### 2.3 降低 Token 消耗与技能干扰

当大量 Skills 和 MCP 同时注入同一会话时，会产生明显副作用：

1. **Token 消耗激增**：每个 Skill 的 SKILL.md 内容、每个 MCP 工具的描述都会进入系统提示，随对话轮次累积。数十个技能 + 数十个工具可轻易占用数万 token，推高调用成本。
2. **技能干扰**：模型需要在众多「何时使用」的说明中做选择，容易误匹配（如把 K8s 问题匹配到 Terraform 技能）、犹豫不决或重复尝试无关工具。
3. **响应质量下降**：上下文过长会稀释关键信息，模型对核心指令的注意力下降，回答易偏离或冗余。

**数字员工的应对方式**：

- **按角色裁剪上下文**：SRE 会话仅加载监控/运维类技能，不加载 coding-agent、terraform 等无关能力，显著减少注入 token 量。
- **减少工具选择负担**：模型只需在 3～5 个相关工具中决策，而非在 20+ 工具中筛选，调用准确率更高。
- **成本可控**：单次对话的 token 消耗与角色复杂度成正比，便于按场景做成本预估与优化。

### 2.4 典型场景

- **SRE 运维专家**：专注 Prometheus 查询、K8s 故障排查、告警分析，配备监控类 MCP 和运维 Skills。
- **K8s 开发助手**：专注清单生成、Deployment/Service 编写，配备 k8s、terraform 等 Skills。
- **DevOps 流水线顾问**：专注 CI/CD、Ansible、Terraform，配备 devops、ansible 等 Skills。
- **内部知识库助手**：专注公司内部文档、运行手册，配备自定义 Skills 和专属 MCP。

通过为不同场景创建数字员工，用户可以在「数字员工」列表中按需选择，点击即可开启与该角色对应的新会话，获得更聚焦、更专业的对话体验。

## 三、配置与使用

### 3.1 入口与列表

在 Control UI 中，进入 **「数字员工」** 标签页，可看到：

- 已创建的数字员工列表
- 每个员工的名称、描述、状态（启用/禁用）、关联技能与 MCP
- 操作按钮：**会话**、**修改**、**禁用/启用**、**删除**

### 3.2 新增数字员工

点击 **「添加」** 打开新增弹框，需填写：

| 字段 | 说明 | 必填 |
|------|------|------|
| **名称** | 展示名称，如「SRE 运维专家」。**名称唯一**，系统会根据名称生成稳定 ID | 是 |
| **描述** | 该数字员工的功能说明，用于列表展示 | 否 |
| **Prompt** | 系统提示词 / 人设说明，定义该角色的语气、风格和职责 | 否 |

**高级配置**（展开后）：

| 字段 | 说明 |
|------|------|
| **MCP 配置** | JSON 格式，与该员工专属的 MCP 服务器配置。会话时会与全局 `mcp.servers` 合并，同 key 时员工配置覆盖全局 |
| **技能名称** | 可选，不填则从上传文件名推导（如 `prometheus-1.0.0.zip` → `prometheus-1.0.0`） |
| **技能文件** | 支持多选，可上传多个 `.md` 或 `.zip` 格式的 Skill 文件，提交时一并上传 |

MCP 配置样例,允许配置多个：
```json
{
  "prometheus_test": {
    "enabled": true,
    "command": "npx",
    "args": [
      "prometheus-mcp@latest",
      "stdio"
    ],
    "env": {
      "PROMETHEUS_URL": "http://192.168.50.59:9090"
    }
  }
}
```

技能文件会存储到 `~/.openocta/employee_skills/<员工ID>/<技能名>/` 下，仅在该数字员工的会话中加载。

> **建议**：每个数字员工配置 2～6 个与该角色强相关的技能即可，避免堆砌过多技能导致 token 消耗增加和模型决策干扰。

### 3.3 修改数字员工

在列表项旁点击 **「修改」**，可编辑：

- **描述**、**Prompt**、**MCP 配置**
- **已有技能**：可标记删除，提交后从员工中移除
- **新上传技能**：可多选新文件，提交后一并上传

**名称不可修改**，以保证会话 key 和技能目录的稳定性。

### 3.4 使用数字员工会话

1. 在数字员工列表中，点击某员工的 **「会话」** 按钮
2. 系统会为该员工创建或复用专属会话（session key 形如 `agent:main:employee:<员工ID>:run:<会话ID>`）
3. 自动跳转到聊天页，并加载该员工的：
   - **Prompt**（人设）
   - **专属 Skills**（`~/.openocta/employee_skills/<员工ID>/` 下的技能）
   - **专属 MCP**（manifest 中配置的 `mcpServers`）
4. 首次进入时，若为新会话，助手会按人设进行简短问候

### 3.5 禁用与删除

- **禁用**：员工仍存在，但列表中显示为「禁用」状态，用户无法开启新会话（已有会话可继续使用，取决于实现策略）
- **删除**：删除该数字员工及其 manifest、专属技能目录，并**同时删除该员工关联的所有会话**（sessions 中的相关条目及 transcript 归档）

## 四、技术说明

### 4.1 数据存储

| 路径 | 说明 |
|------|------|
| `~/.openocta/employees/<员工ID>/manifest.json` | 员工元数据（名称、描述、Prompt、MCP、skillIds 等） |
| `~/.openocta/employee_skills/<员工ID>/<技能名>/` | 该员工专属技能（SKILL.md 及附属文件） |

### 4.2 技能加载优先级（数字员工会话）

数字员工会话的 Skills 按以下顺序加载并合并（后加载覆盖先加载）：

1. **Workspace Skills**（基础池，供 manifest.skillIds 过滤）
2. **embed/agents_skills/<员工ID>**（内置员工技能，若构建时已嵌入）
3. **~/.openocta/employees/<员工ID>/skills**（旧路径，兼容）
4. **~/.openocta/employee_skills/<员工ID>**（新路径，上传的专属技能）

若 manifest 中配置了 `skillIds`，则仅保留 manifest 指定的技能 + 员工专属技能，其余 workspace 技能会被过滤。

### 4.3 MCP 合并规则

会话建立时，MCP 配置合并顺序：

1. 先加载全局 `config.mcp.servers`
2. 若 session key 对应数字员工，再加载该员工的 `manifest.mcpServers`
3. 同 key 时，员工配置覆盖全局配置
